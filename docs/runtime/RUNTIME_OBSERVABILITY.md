# Runtime Observability

## Structured Logging

All logs use `logger` from `src/lib/observability/logger.ts`:

```typescript
logger.info("[component] action", { key: value, correlationId })
logger.warn("[component] warning", { key: value })
logger.error("[component] failure", { error: String(err), correlationId })
```

Convention: prefix with `[subsystem]` for easy filtering:
- `[automation]` — event dispatch and handling
- `[ai/gateway]` — AI inference
- `[job]` — cron job execution
- `[sdk]` — SDK primitive calls
- `[payments]` — payment webhook processing

## Correlation IDs

Every automation event carries a `correlationId` from creation through completion.

Format: `"ae_<13-digit-timestamp>_<4-char-random>"` — generated by `generateCorrelationId("ae")`.

The correlation ID is:
- Stored in `automation_events.payload.correlationId`
- Written to `automation_events.correlation_id` on completion
- Written to `automation_logs.correlation_id`
- Passed through `HandlerContext.correlationId`
- Logged on every job step

To trace a full event lifecycle: query by `correlation_id` across `automation_events` and `automation_logs`.

## Key Database Tables for Observability

### automation_events
Full event lifecycle. Query:
```sql
SELECT * FROM automation_events
WHERE correlation_id = '<id>'
ORDER BY created_at;
```

### automation_logs
Handler execution logs from `logAutomation()`:
```sql
SELECT * FROM automation_logs
WHERE correlation_id = '<id>'
ORDER BY created_at;
```

### workflow_sla_records
SLA tracking per workflow execution:
```sql
SELECT * FROM workflow_sla_records
WHERE correlation_id = '<id>';
```

### ai_generations
AI inference audit log:
```sql
SELECT generation_type, model, tokens_used, was_used
FROM ai_generations
WHERE creator_id = '<creator_id>'
ORDER BY created_at DESC;
```

### ai_cost_events
Per-call cost tracking:
```sql
SELECT agent_name, generation_type, cost_usd, latency_ms
FROM ai_cost_events
WHERE organization_id = '<org_id>'
AND created_at >= now() - interval '30 days';
```

## SLA Tracking

`startSLARecord(workflowId, tenantId, correlationId, eventId)` creates a `workflow_sla_records` row.

`completeSLARecord(slaRecordId, status, error?)` closes it with `completed_at` and duration.

SLA targets come from `workflow_registry.sla_max_ms`:
- WA-01 (WhatsApp received): 30s
- PAY-01/PAY-02 (Payment): 15s
- OB-01 (Onboarding): 30s

## Event Status Dashboard

Query to see event queue health:
```sql
SELECT status, COUNT(*) as count, AVG(attempt_count) as avg_attempts
FROM automation_events
WHERE created_at >= now() - interval '24 hours'
GROUP BY status
ORDER BY count DESC;
```

## Dead Letter Queue

Events in `status = 'dead_letter'` have failed 5+ times. Review `last_error` and `payload` for diagnosis:
```sql
SELECT id, event_name, creator_id, last_error, failed_at, payload
FROM automation_events
WHERE status = 'dead_letter'
ORDER BY failed_at DESC
LIMIT 50;
```

To reprocess a dead letter event, reset:
```sql
UPDATE automation_events
SET status = 'pending', processed = false, processing = false,
    attempt_count = 0, failed_at = null, last_error = null
WHERE id = '<event_id>';
```
